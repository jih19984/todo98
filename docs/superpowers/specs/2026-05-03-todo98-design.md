# Todo98 Design Spec

## Summary

Todo98 is a personal productivity TODO web app with a Retro OS Window / Y2K interface. The product should feel like opening a small Windows 98-era desktop utility, while keeping the actual task workflow modern, fast, and mobile-friendly.

The MVP focuses on finishing a real deployed product with low operating cost: Supabase Free for authentication and data, Vercel Hobby for hosting, and one paid custom domain connected through Vercel. Apple login is explicitly excluded from the MVP. Google and Kakao login are included.

## Goals

- Build a polished personal TODO app that can be used daily by one signed-in user.
- Support real Google and Kakao login through Supabase Auth.
- Store each user's tasks securely in Supabase with row-level security.
- Deploy to Vercel and connect a purchased custom domain.
- Keep the app free to operate except for the domain purchase.
- Make the Retro OS Window visual identity obvious within the first viewport.

## Non-Goals

- No Apple login in the MVP.
- No team collaboration, shared lists, comments, or public profiles.
- No paid subscription, billing, or analytics dashboard.
- No native mobile app.
- No complex project management board in the MVP.

## Product Scope

### Public Landing

The public page presents Todo98 as a nostalgic personal task manager. It should look like a retro desktop with small windows such as `Todo98`, `Today.tasks`, and `Login.exe`.

Primary actions:

- Continue with Google
- Continue with Kakao

The landing page must clearly communicate that the app is a personal TODO tool, not a team project manager.

### Authentication

Authentication uses Supabase Auth providers:

- Google OAuth
- Kakao OAuth

After login, users land directly in the task desktop. If OAuth configuration is missing in local development, the UI should show a clear environment setup error rather than silently failing.

### Task App

The signed-in app contains one main task window with these views:

- Today
- All
- Completed

Users can:

- Create a task
- Edit a task
- Complete or reopen a task
- Delete a task
- Set title
- Set optional note
- Set optional due date
- Set priority: low, normal, high

The MVP should avoid unnecessary complexity. Tags, drag sorting, recurring tasks, and statistics are deferred unless there is spare time after deployment.

### Account and Logout

A small account window or menu shows:

- Signed-in provider/user email when available
- Logout button

## UI Direction

The selected visual direction is Retro OS Window.

Visual rules:

- Windows 98 / early 2000s inspired window chrome.
- Pixel-like borders and title bars.
- Square or lightly rounded controls, not modern pill-heavy UI.
- Bright Y2K accents: electric blue, mint, lemon, pink.
- Avoid heavy dark dashboards for the default experience.
- Use OS-like labels such as `Today.tasks`, `NewTask.exe`, `Settings.exe`.
- Keep readability and accessibility higher priority than nostalgia.

The interface should still feel usable on mobile. On small screens, windows stack vertically and keep touch targets large enough to tap comfortably.

## Data Model

### `tasks`

Fields:

- `id`: UUID primary key
- `user_id`: UUID, references Supabase auth user
- `title`: text, required
- `note`: text, nullable
- `due_date`: date, nullable
- `priority`: text, one of `low`, `normal`, `high`
- `completed_at`: timestamp, nullable
- `created_at`: timestamp
- `updated_at`: timestamp

Indexes:

- `user_id`
- `user_id, due_date`
- `user_id, completed_at`

Security:

- Enable row-level security.
- Users can select, insert, update, and delete only rows where `auth.uid() = user_id`.

## Architecture

### Frontend

Use Next.js with TypeScript.

Core areas:

- Auth client setup for Supabase.
- Landing/login page.
- Auth callback handling.
- Protected task app page.
- Task CRUD service.
- Retro OS component primitives: window, title bar, button, input, modal.

### Backend

Use Supabase:

- Auth providers: Google and Kakao.
- Postgres database.
- Row-level security policies.

No custom backend server is required for the MVP.

## Environment Variables

Required app variables:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

Supabase provider setup requires OAuth credentials in the Supabase dashboard:

- Google client ID and secret
- Kakao REST API key and client secret if enabled

Vercel must receive the same public Supabase environment variables.

## Deployment and Domain

Deployment target:

- Vercel Hobby

Domain flow:

1. Build and deploy the app to Vercel.
2. Buy a custom domain from a registrar.
3. Add the domain in Vercel project settings.
4. Update DNS records as Vercel instructs.
5. Add the production domain to Supabase auth redirect URLs.
6. Add the production OAuth redirect URLs to Google and Kakao developer consoles.
7. Verify Google and Kakao login on the production domain.

## Error Handling

Authentication:

- Show a clear message when OAuth login fails.
- Show a setup message when Supabase environment variables are missing.

Tasks:

- Empty title cannot be submitted.
- Failed task operations show a compact retro dialog.
- Optimistic UI is optional; correctness matters more than animation.

## Testing

Use TDD for implementation.

Test coverage should include:

- Task filtering: Today, All, Completed.
- Task creation validation.
- Task completion and reopening.
- Task update and deletion service behavior.
- Login buttons call the correct Supabase OAuth provider.
- Protected app redirects or blocks unauthenticated users.

Manual verification should include:

- Mobile viewport.
- Desktop viewport.
- Google login on production.
- Kakao login on production.
- Vercel custom domain loads the app over HTTPS.

## Open Decisions

None. The MVP excludes Apple login and paid app infrastructure. Domain purchase is the only expected paid item.
